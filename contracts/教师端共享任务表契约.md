# 教师端共享任务表契约

状态：唯一当前契约  
日期：2026-07-23

## 1. 怎么交换

任务触发中心与教师端共用同一个 PostgreSQL：

- `task_templates` 是唯一任务目录，两端只读已发布内容；
- `task_assignments` 是唯一任务实例与当前状态事实；
- 不建任务发布 HTTP 接口、状态回传 HTTP 接口或第二份任务副本。

共库不等于共用超级账号。教师浏览器不直连数据库；教师端后端使用 `tit_teacher_crud`，任务触发中心使用独立账号。

## 2. 谁创建、谁更新

| 方向 | 写入内容 | 所有权 |
|---|---|---|
| 任务触发中心 → 共享库 | 新教师出现时初始化 G01–G10 固定成长 assignment | `FIXED_GROWTH + TRIGGER_CENTER` |
| 任务触发中心 → 共享库 | 按确定性规则创建 5 类个性化 assignment | `PERSONALIZED_IMPROVEMENT + TRIGGER_CENTER` |
| 教师端 → 共享库 | 更新已有 G/P 任务的执行状态 | 只更新状态五字段 |
| 任务触发中心 ← 共享库 | 读取 G01–G10 的合法 `COMPLETED` 并幂等结分 | 不修改教师完成结论 |

固定任务不是人工发布：教师首次进入试用期教师集合时，由任务触发中心自动建立 10 条共享任务事实。初始化本身不创建通知、提醒或投递意图。个性化任务由触发中心按已确认规则创建。教师端直接读取已有 G/P 行并负责“接收→完成”，不再创建任务。

### 3.3 固定成长积分与出营资格

- G01–G10 的当前完成状态只认共享 `task_assignments.status`。除 `COMPLETED` 外，其他状态均不加该任务分。
- 每个 assignment 通过 `template_version_id` 固定引用一条模板记录；合法完成后按该记录的 `payload.score_value` 累加。已经完成的任务立即进入当前成长任务分，不必等待 10 项全部完成。
- 当前成长任务分为已完成 G01–G10 的模板分值之和，最高 30 分。教师端不能写 `score_value`、积分流水、总分或资格结论。
- 出营资格中的“必修成长任务全部完成”要求同一教师的 G01–G10 共 10 条 assignment 全部存在且全部为 `COMPLETED`。初始 `ASSIGNED` 明确表示“任务已建立、尚未完成”；少于 10 条属于内部 `TASK_BASELINE_INCOMPLETE`，不能解释为教师未完成，也不能标为外部 `SOURCE_MISSING`。
- 当前出营另外两项硬门槛为 L0 投诉 0 次、raw 总分不低于 100。金牌资格只有两项硬门槛：已满足出营资格、raw 总分不低于 200。

## 4. `task_templates` 共享字段

| 字段 | 含义 | 为什么必须有 |
|---|---|---|
| `row_id` | 模板记录 ID | assignment 冻结引用的外键 |
| `template_id` | G/P 稳定任务编码 | 两端使用同一业务类型 |
| `template_version` | 模板序号 | 识别 assignment 创建时所用内容 |
| `status` | `DRAFT / PUBLISHED / RETIRED` | 只允许已发布模板创建任务 |
| `revision` | 编辑乐观锁 | 防止配置页并发覆盖 |
| `output_type` | 输出类型，当前为 `TEACHER_TASK` | 避免把 Case/提醒当成教师任务 |
| `execution_owner` | 任务执行方，当前为 `TEACHER_APP` | 明确“收到→完成”归教师端 |
| `integration_mode` | `INBOUND_STATUS_ONLY / OUTBOUND_MANAGED` | G 只接收教师端状态更新；P 由触发中心按规则管理 |
| `external_task_template_code` | 教师端执行页路由码 | 让教师端稳定打开对应流程 |
| `source_mode` | 模板来源性质 | 区分真实配置与测试种子 |
| `payload` | 静态标题、Why、How、完成标准、分值、截止规则 | 静态内容只存一份 |
| `created_by / updated_by` | 创建与更新主体 | 审计追溯 |
| `created_at / updated_at` | 创建与更新时间 | 判断内容的时间边界 |

## 5. `task_assignments` 共享字段

| 字段 | 含义 | 为什么必须有 |
|---|---|---|
| `assignment_id` | 任务实例 ID | 两端定位同一条任务 |
| `teacher_id` | 任务所属教师 | 决定谁能看到和执行 |
| `task_code` | G/P 稳定任务编码 | 选择正确的教师端执行流程 |
| `template_version_id` | `task_templates.row_id` | 冻结本实例使用的静态内容 |
| `task_kind` | `FIXED_GROWTH / PERSONALIZED_IMPROVEMENT` | 区分加分的固定成长与不加分的改善任务 |
| `creator_system` | 当前为 `TRIGGER_CENTER` | 强制任务创建所有权；教师端不能创建 assignment |
| `status` | 当前执行状态 | 两端共享的核心事实 |
| `priority` | `P0 / P1 / P2 / P3` | 教师端和运营端一致排序 |
| `why` | 该教师为什么收到本任务 | 必须向教师解释触发原因 |
| `display_title` | 本实例展示标题 | 支持“一般投诉-[三级分类]问题”等动态名称 |
| `evidence_snapshot` | 触发时最小证据快照 | 教师端可解释，运营可追溯，且不依赖后续源数据变化 |
| `due_at` | 截止时间 | 排期和超时判定 |
| `timezone_used` | 截止时间所用时区 | 避免跨时区任务歧义 |
| `timezone_source` | 时区来源 | 追溯时区是教师资料、人工或默认值 |
| `timezone_verified_at` | 时区核验时间 | 证明截止时间计算依据 |
| `status_reason_code` | 失败、过期、豁免或取消的稳定原因码 | 让异常状态可统计，不只留自由文本 |
| `source_mode` | 本实例的数据性质 | 不把 Mock 当真实任务 |
| `dedupe_key` | 任务幂等键 | 防止重试或重复批次创建同一任务 |
| `created_by` | 创建服务/账号 | 创建审计 |
| `updated_by` | 最后状态更新服务/账号 | 更新审计 |
| `row_version` | 乐观锁序号 | 防止旧状态覆盖新状态 |
| `assigned_at` | 任务成为可读事实的时间 | 不用教师首次看到时间代替发布时间 |
| `status_changed_at` | 当前状态生效时间 | 状态时效和审计 |
| `completed_at` | 可信完成时间 | 只有 `COMPLETED` 可非空，用于结分/效果分析 |
| `created_at` | 数据库建行时间 | 技术追溯 |
| `updated_at` | 最后合法更新时间 | 同步与排错 |

`due_at` 为空时三个时区证据字段必须同时为空；有截止时间时必须同时完整。

## 6. 具体写法

### 6.1 任务触发中心初始化 G01–G10

触发条件只有一个：新教师首次写入 `teachers`，即系统首次识别到该教师进入试用期培养范围。同一教师重复同步、接口重试或补数不能重复创建。

初始化必须满足：

- 一次事务内引用当前已发布的 G01–G10，恰好创建 10 条；
- `task_kind = FIXED_GROWTH`；
- `creator_system = TRIGGER_CENTER`；
- `dedupe_key = fixed:{teacher_id}:{task_code}`；
- 初始 `status = ASSIGNED`，业务含义是“已生成、未完成”；
- 不写 `notifications`，不创建提醒或投递意图；
- 目录不是恰好 10 项、模板引用无效或最终少于 10 条时失败关闭，并记录内部异常 `TASK_BASELINE_INCOMPLETE`。

固定任务初始化必须幂等。已有合法 assignment 保留原状态，只补缺失项，不覆盖教师端已经写入的进度或完成事实。

### 6.2 任务触发中心创建个性化任务

触发中心只能引用已发布的 5 个 P 模板，并写入：

- `task_kind = PERSONALIZED_IMPROVEMENT`；
- `creator_system = TRIGGER_CENTER`；
- 可读的 `why` 和 `display_title`；
- 只含必要证据的 `evidence_snapshot`；
- 与触发范围一致的 `dedupe_key`。

教师端只消费共享行，不改 Why、标题、证据、优先级、截止时间或模板引用。

### 6.3 教师端更新所有已有任务

教师端不拥有 `INSERT` 或 `DELETE` 权限，只可更新已有 assignment 的：

`status, status_reason_code, status_changed_at, completed_at, updated_by`

更新必须在 `WHERE` 中带旧 `row_version`：

```sql
UPDATE task_assignments
SET status = :new_status,
    status_reason_code = :reason_code,
    status_changed_at = :changed_at,
    completed_at = :completed_at,
    updated_by = :service_name
WHERE assignment_id = :assignment_id
  AND row_version = :expected_row_version;
```

影响行数为 0 时先重读，不盲目重试覆盖。数据库会自动递增 `row_version`、更新 `updated_at`、写审计，并在状态变化时写 Outbox。

## 7. 状态机

状态集：

`ASSIGNED / VIEWED / IN_PROGRESS / SUBMITTED / UNDER_REVIEW / COMPLETED / FAILED / EXPIRED / WAIVED / CANCELLED`

- `VIEWED / IN_PROGRESS / SUBMITTED` 来自教师真实动作，不由时间推断。
- `COMPLETED / FAILED / EXPIRED` 由教师端可信服务按模板完成标准产生。
- `WAIVED / CANCELLED` 只由有权运营或系统流程产生，教师端受限角色不能写。
- `COMPLETED / EXPIRED / WAIVED / CANCELLED` 是不可回退终态；`FAILED` 可进入重试或完成。
- `FAILED / EXPIRED / WAIVED / CANCELLED` 必须有 `status_reason_code`；`COMPLETED` 必须有 `completed_at`。
- assignment 不允许删除。误报完成不能直接回退，必须走受控修正与审计。

## 8. 生产接入门

当前共享表、字段权限和数据库约束已在本地实现。生产还必须完成：

1. 教师端生产后端使用独立受限账号，不使用表所有者或超级用户。
2. 验证新教师写入后自动出现恰好 10 条 `ASSIGNED` 固定任务，重复同步不重复创建，且初始化不产生通知。
3. 验证教师端不能创建或删除 assignment，只能合法更新已有 G/P 状态；同时验证终态拒绝和旧 `row_version` 并发更新失败。
4. 验证教师端不能改 `priority / why / display_title / evidence_snapshot / due_at / template_version_id`。
5. 验证每次创建/更新均有 `audit_events`，状态变化均有 `outbox_events`；固定任务初始化不得创建 `notifications`。
6. 验证同一 G assignment 重复完成不会重复加分。
7. 监控每位教师固定任务数量必须等于 10；偏差统一记为 `TASK_BASELINE_INCOMPLETE`，不得归为外部数据缺失。

通过以上验收前，只能称为“本地共享表契约可运行”，不能称为“教师端生产已打通”。
