# 数据与积分规则

日期：2026-07-23  
状态：唯一当前口径

## 1. 事实分层

- `data_import_batches` 是每次数据接入的批次头。一次性文件使用 `MANUAL_BASELINE`，未来每日接口使用 `API_DAILY`。
- `source_records` 无损保存课程等外部原始行；不用类型化投影代替原始证据。
- `teacher_metric_snapshots` 保存教师维度统计快照，是当前积分公式的输入事实。
- `lesson_facts` 是课程级查询投影，当前第一批课程源的 22 个字段全部保留；本批先用于个性化触发，不与教师快照重复计分。
- `complaint_category_rules` 保存投诉三级分类精确匹配规则。原始 P0–P4 不改写，同时保存 `severity_rank=0–4`。
- `personalized_trigger_matches` 保存规则命中、最小证据和最终输出的追溯关系。
- `task_assignments` 保存固定成长与个性化改善任务的当前状态；只有 G01–G10 的合法 `COMPLETED` 可产生成长积分。
- `score_entries` 是不可变积分流水和结算审计。固定成长任务的当前计分事实仍以共享 `task_assignments` 的状态及其固定引用的模板分值为准，教师端不直接写分。

教师快照中的 `peak_slot_cnt`、`first_booked_date`、`is_cpl_tesol`、`is_self_introduce` 均是类型化字段。生产计算读类型化字段，`raw_payload` 只用于回查。

## 2. 数据状态

| 值 | 含义 |
|---|---|
| `REAL` | 上游直接提供的事实 |
| `DERIVED_REAL` | 只由真实字段和确定公式派生 |
| `MOCK` | 测试数据 |
| `MOCK_SIMULATION` | 基于真实规模的模拟结果 |
| `MOCK_PROXY` | 暂用相邻字段代替目标事实 |

没有证据时对应分数为 `0`，但必须另记 `SOURCE_MISSING`；`0` 不得伪装成“已确认没有问题”。

## 3. 积分与资格

五个维度独立累计，不使用权重、维度封顶或维度最低线：

```text
用户反馈分 = 好评次数 × 5 + 收藏人数 × 5 + 15 日复约人数 × 8
准时完课数 = max(完课数 - 迟到次数 - 早退次数, 0)
可靠性分 = 准时完课数 × 2 + Peak 完课数 × 1
课堂质量分 = perfect_cnt × 1.6
供给分 = peak_slot_cnt 首次达到 40 时 +10，之后锁定
固定成长任务分 = Σ（状态为 COMPLETED 的 G01–G10 assignment 所固定模板的 score_value），最高 30
raw 总分 = 五个维度之和
显示分 = min(raw 总分, 200)
```

供给分不是任务，不进 `task_assignments`。5 个个性化改善任务均为 0 分。
`perfect_cnt` 是教师宽表中的完美完课数，当前课堂质量分直接按该真实字段计算，
不再读取 `class_quality_no_issue_rate`，也不使用 80% 模拟达成率。字段缺失时仍按
“无证据得 0 分”处理并标记 `SOURCE_MISSING`。

固定成长任务不读取教师快照中的 `new_teacher_task_score` 作为当前结论。每个
G01–G10 assignment 通过 `template_version_id` 固定所用模板；状态合法变为
`COMPLETED` 后，立即按该 `task_templates.payload.score_value` 计入当前成长任务分，
并同步更新当前教师快照的 `new_teacher_task_score`、`raw_total_score` 和
`public_total_score`，不必等待其余任务完成。任务状态与模板分值仍是事实源，
上述字段只是可查询投影。新教师首次进入系统时由本系统初始化 G01–G10，默认状态为
`ASSIGNED`（未完成）。少于 10 条 assignment 属于内部 `TASK_BASELINE_INCOMPLETE`，
不能解释为教师未完成，不能标为外部 `SOURCE_MISSING`，也不能用默认 30 分补齐。

出营必须同时满足：

- G01–G10 共 10 个固定成长 assignment 全部存在且状态均为 `COMPLETED`；
- L0 投诉次数为 `0`；
- `raw_total_score >= 100`。

L0 投诉次数按 `lesson_facts.complaint_level_rank = 0` 聚合。存在投诉但三级分类尚未
精确映射出级别时，投诉证据视为不完整，资格门禁失败关闭，不能当作 0 次 L0。

金牌必须且只需同时满足：

- 已满足上述出营资格；
- `raw_total_score >= 200`。

基础分、30 天完课量、用户反馈分、迟到、早退和真实缺席不再是额外的金牌硬门槛。

分数线命中、任务完成、积分结算、资格达成和后续业务效果是不同事实，不能互相代替。

### 3.1 历史规则兼容边界

- `v6` 是当前生效口径：保持 `perfect_cnt × 1.6` 和三个出营硬门槛，并将金牌硬门槛收敛为“满足出营资格 + raw 总分不低于 200”。
- `v2`–`v5` 仅保留用于读取、审计历史快照；其中旧课堂质量达成率、`severe_redline_event`、旧出营门槛和 v5 的额外金牌门槛均不得参与当前资格判断。
- 历史配置记录不原地改写。当前页面、重算和新快照只使用已发布的 `v6`。

## 4. 个性化触发口径

规则是确定性 Workflow，不由模型猜测。同一课程按以下顺序评估：

1. **Lesson Memo 优先**：`absence_reason_detail = "Unfilled Lesson Memo"` 时创建 `P-REL-MEMO`，展示名“出席（未填写lesson-memo）问题”。同课的迟到/早退/假早退只放进证据说明，不再重复发出席任务。
2. **出席问题**：缺席原因非空且不是 Lesson Memo，或迟到/早退/假早退任一为 1，创建 `P-REL-ATTENDANCE`，教师完成培训和 quiz。
3. **投诉路由**：先用投诉三级分类精确匹配 `complaint_category_rules`。
   - 投诉二级分类为“出席问题”，改走可靠性出席任务；
   - 投诉二级分类为“网络设备问题”，改走课堂质量提醒；
   - 其他投诉的 P0/P1（`severity_rank 0/1`）创建运营 Case，标题“严重投诉-[投诉三级分类]问题”；
   - 其他投诉的 P2/P3/P4（`severity_rank 2/3/4`）创建 `P-FB-COMPLAINT`，标题“一般投诉-[投诉三级分类]问题”；
   - 三级分类无精确映射时记 `PENDING_DATA`，不自动输出。
4. **重复差评标签**：只读取 `差评标签 = 1` 的课程，将 `评价详情` 按英文逗号拆分、去除首尾空格并在单课内去重。同一教师的同一标签出现在至少 2 个不同课程 ID 时，创建一条 `P-FB-NEGATIVE`，标题“差评-[标签名称]问题”；命中次数按不同课程数计算。
5. **拉黑聚合**：同一教师被至少 2 个不同学员拉黑时，首次达阈值创建 `P-FB-BLACKLIST`，标题“拉黑问题”。
6. **课中质量**：未开摄像头、CPU 过高或网络延迟过高任一为 1，创建 task-less 站内提醒，标题“课中质量问题”，说明具体异常。

当前 2.0 基线共有 243 节差评课，均能从评价详情解析出标签；共形成 61 个“教师 × 重复标签”任务，涉及 33 位教师。没有评价详情缺口时，不再生成差评 `PENDING_DATA`。

严重投诉是运营 Case，课中质量是提醒，都不伪装成教师任务。每个命中用 `dedupe_key` 幂等，并在 `personalized_trigger_matches.evidence_snapshot` 保留当时证据。

## 5. 导入与重算安全

- 导入前校验 Sheet、字段顺序、行完整性、教师 ID、业务键和文件摘要；任一失败都不提交半批数据。
- 课程原始行、投诉规则、类型化投影和触发输出在同一事务中落库。
- 相同 `source_sha256 + source_sheet` 重复导入必须幂等；同一课程 ID 不能跨批次静默覆盖。
- 历史原始行、快照、规则命中和积分流水不原地改写；纠错通过新批次、修正事实或冲正流水完成。
- `source_records.raw_payload` 可含原始学员 ID，不对运营前端返回；前端只使用 `student_id_hash` 的聚合结果。
